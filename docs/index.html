01

Fetching from api-tennis

The source is api-tennis.com. A single method, get_livescore, returns every match currently in play. The API is query-string driven and wants the key in the URL, not a header:

// TennisApiClient — one call per cycle GET https://api.api-tennis.com/tennis/?method=get_livescore&APIkey=<KEY>

The response is deserialized into LiveScoreEnvelope { success, result[] }. If success != 1 we return an empty list rather than throwing — one bad poll never takes the service down. Each element of result[] is one match in raw, snake-cased, everything-as-string form.

The feed's non-obvious realities

Most of the engineering effort lives here, because the feed has sharp edges that will silently corrupt the data if taken at face value:

02

Mapping — the anti-corruption layer

The mapper is the boundary between the messy feed and everything downstream. Its output, MappedMatch, is a clean object the rest of the system trusts. Three transformations matter most.

Set scores

The games count is the integer before the decimal ("7.7" → 7 games; the .7 is the tie-break mini-score). The set number comes from the feed's score_set field, so a skipped or oddly-encoded set never renumbers the others.

Point winners

A game score is a ladder of tokens 0 / 15 / 30 / 40 / A. The winner of each point is inferred from which side's token advanced; when advantage drops back to 40, the point went to the opponent. If a winner can't be derived, that point is dropped rather than guessed.

Statistics

Only the match-total period is kept. Everything is normalised into a MatchStatLine that can present itself as a ratio (29/36), a percentage ("81%"), or a bare count ("14 aces").

03

Storing — eight tables, one transaction

The sync is a background service on a 25-second PeriodicTimer. Each tick opens a fresh dependency scope and runs a strictly diff-based reconciliation: upsert tournaments, players and the match, then append only the games, points and stats that are missing. Nothing is ever deleted — a match that leaves the feed is simply marked IsFinished.

A subtle EF detail that matters. Every primary key is a Guid assigned in code. Left at EF's default (store-generated), a new child added to an existing match is mistaken for an existing row and issued as an UPDATE that affects zero rows — which fails the entire save the moment a live match gains a new game or point. All Guid keys are therefore marked ValueGeneratedNever, so new children are correctly inserted. This is why the sync must never run as two instances at once against the same database.
04

The math of momentum

Momentum is a path-dependent fold: each point carries a signed weight that is added to the running total of whoever won it. That is the entire model — a deterministic sum over the ordered timeline, with no clock and no randomness.

+2.0

+2.0

−2.0
Double fault (server)

A single branch fires per point — signals never double-count. Winning on return is worth twice a routine hold because it threatens the opponent's serve; a break, resolved at the game boundary from the reliable serve_winner, is the strongest in-game signal.

The fold

// for each point, in order cumulative[winner] += weight ewma[winner] = 0.25·weight + 0.75·ewma_prev // "who's hot now"

The cumulative line drives the main graph; the EWMA is a smoothed "surge meter" that reacts to the last handful of points. The break bonus is emitted as its own snapshot at the game boundary, so a break still registers even for a game with no point detail.

Momentum is a trend, not the score. A player who holds comfortably early accumulates a real edge before any break happens. The graph tracks who is winning the recent exchanges; it is designed to lead — and sometimes disagree with — the games count.
05

The math of win probability

A classic hierarchical model. One number — the chance of winning a point on serve — is propagated upward through game, set and match. Every level is analytic and memoised, so a full estimate is cheap enough to run on every score change.

🏆 Match

Match(sa, sb) = pSet · Match(sa+1, sb) + (1 − pSet) · Match(sa, sb+1)
▲ built from ▲

Set

Set(ga, gb) = winGame · Set(ga+1, gb) + (1 − winGame) · Set(ga, gb+1) · 6-6 → tie-break
▲

Game

Game(a, b) = p · Game(a+1, b) + q · Game(a, b+1) · deuce = p² / (p² + q²)
▲

Point

p = probability of winning a point on your own serve · default 0.62

The current game and set use the live score; not-yet-started sets use a serve-neutral estimate. The final answer combines the set in progress with the remaining match:

P(A wins) = currentSet · Match(setsA+1, setsB) + (1 − currentSet) · Match(setsA, setsB+1)
What the model does and doesn't know. With p = 0.62 for both players, the estimate is purely score-driven — it has no notion of ranking or form. A player up a set and 5-0 reads ~99.9% from the scoreline alone, shown in the UI as >99% rather than a misleading absolute 100%. The hook to make it player-aware already exists: feed real serve-hold rates into FirstServePointWin / SecondServePointWin instead of the flat default.
